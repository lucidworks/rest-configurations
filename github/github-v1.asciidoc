== GitHub REST Configuration

This documentation describes aspects of the GitHub REST configuration such as the authentication methods, endpoints requested, data crawled, pagination information. Terminology is also provided as a reference.

The JSON template file `github-repo-v1.json` crawls a single specific repository via `/repos/{owner}/{repo}`. To crawl multiple repositories, create one datasource per repository.

The GitHub REST Configuration will index each GitHub object listed below, as a separate solr document:

* Repositories
* Issues
* Pull Requests
* Commits
* Commit Diffs (per-commit file changes and stats)
* Branches
* Tags
* Milestones
* Collaborators
* Releases
* Issue Comments
* Pull Request Review Comments
* Commit Comments
* Content (root directory listing)
* Folders (recursive directory traversal)
* Blobs (file content via Contents API binary parsing)

The Configuration is based on the Discovery of objects at Hierarchical Requests, supported since plugin version rest-1.1.0

The configuration uses the GitHub REST API. The endpoints do not explicitly specify a version; they default to `2022-11-28`. See https://docs.github.com/en/rest/about-the-rest-api/api-versions[API Versions | GitHub Docs^] for details.

The configuration was tested with GitHub Cloud (github.com). For GitHub Enterprise, update the `serviceURL` property to point to the enterprise instance API URL, e.g. `https://github.example.com/api/v3`.

== Authentication methods

Basic Authentication using the GitHub username and a Personal Access Token (PAT) as the password. For more information, see link:https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/managing-your-personal-access-tokens[Managing your personal access tokens | GitHub Docs^].

=== Classic Personal Access Token (PAT)

For **public repositories only**, most endpoints require no additional scopes. However, crawling **private repositories** or **organization-private repositories** requires the following scopes:

* `repo` - Full control of private repositories (grants access to all repository data)
* `read:org` - Read organization membership (required for organization-private repositories and the collaborators endpoint)

**Minimum recommended scopes:**

* Public repositories only: `public_repo` + `read:org`
* Private repositories: `repo` + `read:org`

=== Fine-Grained Personal Access Token

Fine-grained personal access tokens are also supported. Set the token's repository access to target the desired repositories, then grant the following **read-only** permissions:

* **Metadata** (read) - Required for repository listing, collaborators, tags, and commit comments
* **Contents** (read) - Required for commits, branches, and releases
* **Issues** (read) - Required for issues, milestones, and issue comments
* **Pull requests** (read) - Required for pull requests and pull request review comments

NOTE: The **collaborators** endpoint is an outlier: in addition to the scopes listed above, the authenticated user must have **push (write), maintain, or admin** access to the repository. Without this level of access, the endpoint returns HTTP 403 regardless of token scopes. If the crawl account does not have write access to all repositories, consider removing the COLLABORATOR request configuration from the JSON to avoid 403 errors.

NOTE: Draft releases are only visible to users with push (write) access to the repository.

=== Rate limit impact

Unauthenticated requests are limited to 60 requests per hour. Authenticated requests (with any valid PAT) are allowed 5,000 requests per hour. Always use an authenticated token to avoid rate limiting during crawls.

== Supported crawl options

* Full crawl:
** All the content from the source is fetched.

* Re-Crawl:
** Per re-crawl, all the content from the source is retrieved as it were a full-crawl.
** Orphan objects (deleted in the GitHub source that are not retrieved with a current crawl), will be deleted from the index using the strayContentDeletion feature from connectors-service, which is run when a crawl finishes.

== Rate Limiting

GitHub enforces a rate limit of 5,000 authenticated requests per hour. Retry properties (`retryCount`, `maxDelayTime`) can be configured under the datasource's retry settings to provide resilience against transient errors and rate limit responses (HTTP 403/429), though the default values are typically sufficient.

For repositories with large amounts of data, consider the total request volume: each repository triggers up to 15 child requests (one per entity type), and each child request paginates at 100 items per page.

NOTE: The CONTENT, FOLDER, and BLOB requests use the GitHub Contents API (`/repos/{owner}/{repo}/contents/{path}`) to crawl repository file content. Unlike the previous Git Trees/Blobs approach, this does not produce 404 errors since the Contents API natively handles both files and directories. However, each directory level requires a separate API call (one request per directory). File downloads via `/contents/{path}` with the `Accept: application/vnd.github.raw+json` header count against the API rate limit. For large repositories with many directories and files, this can consume the 5,000 requests/hour rate limit. Consider removing the CONTENT, FOLDER, and BLOB request configurations from the JSON if file content indexing is not needed.

For GitHub Enterprise instances, rate limits may differ. Consult your administrator.

== Pagination Setup

Pagination by Batch Size is configured per child request that returns paginated arrays, using a page-number approach. The following child requests use pagination: ISSUE, PULL_REQUEST, COMMIT, BRANCH, TAG, MILESTONE, COLLABORATOR, RELEASE, ISSUE_COMMENT, PR_REVIEW_COMMENT, COMMIT_COMMENT.

GitHub REST API uses page-based pagination with `page` and `per_page` query parameters. The API returns bare JSON arrays. When there are no more results, an empty array `[]` is returned.

=== Configure the 'Pagination By BatchSize' properties:

* `IndexStart: 1` - GitHub pages are 1-indexed (first page is page 1)
* `BatchSize: 1` - Used to increment the page number by 1 each iteration. The `${LW_INDEX_START}` variable produces values 1, 2, 3, etc.
* `Stop Condition Key: $` - References the root response (bare JSON array)
* `Stop Condition Value: []` - Pagination stops when the response is an empty array

=== Configure Query Params:

* `per_page=100` - The maximum number of items GitHub returns per page
* `page=${LW_INDEX_START}` - The current page number, auto-incremented by the connector

NOTE: The `batchSize=1` setting is a technique to generate sequential page numbers (1, 2, 3...) from the `${LW_INDEX_START}` variable, since GitHub uses page-number pagination rather than offset-based pagination. The actual number of items per page is controlled by the fixed `per_page=100` query parameter.

== Variables used

The GitHub REST configuration variables used are:

* `${LW_INDEX_START}` - Used with pagination feature. This variable is used to set the `page` query parameter, which is the page number to retrieve. GitHub pagination is 1-indexed. The connector increments this value by `batchSize` (1) after each page request, producing page numbers 1, 2, 3, etc.

* `${LW_PARENT_DATA_KEY}` - Used with Child Request Configuration. This variable is replaced with the value of the `parentIdKey` field from the parent object's response. The connector requires `parentIdKey` on all child requests. In the repo template, most children set `parentIdKey=full_name` (required by the connector) but hardcode the `{owner}/{repo}` path in the endpoint rather than using `${LW_PARENT_DATA_KEY}`. The CONTENT request sets `parentIdKey=path` so its FOLDER child receives the file/directory path via `${LW_PARENT_DATA_KEY}` in its endpoint (`/contents/${LW_PARENT_DATA_KEY}`). The FOLDER request also sets `parentIdKey=path`, enabling recursive directory traversal and passing the path to its BLOB child. The BLOB request uses `${LW_PARENT_DATA_KEY}` in its endpoint to download file content from `/contents/${LW_PARENT_DATA_KEY}`. Similarly, the COMMIT request sets `parentIdKey=sha` so the COMMIT_DIFF child receives the commit SHA via `${LW_PARENT_DATA_KEY}` in its endpoint (`/commits/${LW_PARENT_DATA_KEY}`). In the org template (`github-org-v1.json`), all children use `${LW_PARENT_DATA_KEY}` in endpoints with `parentIdKey=full_name` to resolve the `owner/repo` path dynamically.

=== Notes

* **Single-repo crawling**: The template uses `/repos/{owner}/{repo-name}` to target a single repository. Replace the placeholders in the endpoint with the target owner and repo name. To crawl multiple repositories, create one Fusion datasource per repository using this template.

* **objectType uniqueness constraint**: The `requestConfigurations` pattern enforces that each `objectType` value must be unique across all request blocks. This means you cannot duplicate the root REPOSITORY block to crawl multiple repos in a single datasource. Attempting to do so will result in only the last REPOSITORY block being processed. To crawl multiple specific repos, create one datasource per repo.

* The requests are linked hierarchically by using the properties *ObjectType and ParentObjectType*.
** Most child requests use REPOSITORY as their Parent ObjectType with `parentIdKey=full_name` (required by the connector for all child requests). In the repo template, child endpoints hardcode the `{owner}/{repo}` path — the `parentIdKey` is still set because the connector requires it, but the value is not used in endpoint construction. The CONTENT request sets `parentIdKey=path` so its FOLDER child receives the file/directory path via `${LW_PARENT_DATA_KEY}`. FOLDER also sets `parentIdKey=path` so its BLOB child receives the file path via `${LW_PARENT_DATA_KEY}`. This creates a CONTENT → FOLDER → BLOB chain using `parentIdKey=path` throughout. Similarly, the COMMIT request sets `parentIdKey=sha` so its COMMIT_DIFF child receives the commit SHA via `${LW_PARENT_DATA_KEY}`.
** When objects are indexed, the field `_lw_rest_parent_object_ss` keeps the list of parents related to an object.

* GitHub's `/repos/{owner}/{repo}/issues` endpoint returns both issues and pull requests. PRs can be identified by the presence of the `pull_request` key in the response object. The separate PULL_REQUEST request (`/pulls` endpoint) provides additional PR-specific fields. This means PRs appear in both ISSUE and PULL_REQUEST results with different field sets.

* Comment endpoints use repo-level listing (`/issues/comments`, `/pulls/comments`, `/comments`) rather than per-issue or per-commit endpoints. This design avoids the need for nested parent key substitution. The parent entity can be identified using fields within each comment:
** Issue comments: `issue_url` field
** PR review comments: `pull_request_url` field
** Commit comments: `commit_id` field

* The COMMIT_DIFF request re-fetches each commit via the single-commit detail endpoint (`/repos/{owner}/{repo}/commits/{sha}`) to obtain the `files` array and `stats` object, which are not included in the list commits response (`/repos/{owner}/{repo}/commits`). This means each commit generates one additional API call. For repositories with many commits, this can significantly increase the total request volume and rate limit consumption.

* GitHub API responses are bare JSON arrays (e.g., `[{...}, {...}]`) rather than wrapped objects (e.g., `{"items": [...]}`). The `dataPath` is set to an empty string to handle this. If the connector does not correctly parse bare arrays, try `dataPath: "$"` as an alternative.

* **CONTENT, FOLDER, and BLOB requests**: The configuration uses the GitHub Contents API (`/repos/{owner}/{repo}/contents/{path}`) to crawl repository file content, following a recursive directory traversal pattern similar to the Alfresco connector. The CONTENT request lists the root directory via `/repos/{owner}/{repo}/contents`, returning entries for each file and directory. The FOLDER request recursively walks subdirectories using `recursiveRequest=true` — each directory entry triggers a new request to `/repos/{owner}/{repo}/contents/{path}`, which returns the directory's children. The BLOB request downloads raw file content from the same `/contents/{path}` endpoint using the `Accept: application/vnd.github.raw+json` header with `binaryResponse=true`, enabling Tika-based text extraction. Both CONTENT and FOLDER use `skipIndexation=true` — they exist only for discovery and do not create Solr documents. Only BLOB entries are indexed. This approach produces no 404 errors since the Contents API natively handles both files and directories. GitHub returns both file and directory entries in the same response; the connector handles them via the FOLDER/BLOB child request split.

NOTE: Each directory level requires a separate API call. A deeply nested repository will generate more API requests than a flat one. However, this approach avoids the 404 errors that occurred with the previous Git Trees/Blobs approach and is consistent with how the classic GitHub connector works.

== Response Parsing Configuration

Per request, configure the property *Response Handling* to set up how to parse the response (`responseConfiguration` in the JSON files)

=== Plugin Parsing:

* This parsing happens by default. The responses are parsed as a JSON Object structure using JsonPath.
* Plugin Parsing will happen for all the requests listed in the Table 'Endpoints Configuration with GitHub REST Connector', except the BLOB request which uses binary parsing. The CONTENT and FOLDER requests also use plugin parsing but with `skipIndexation=true` — they parse the JSON response to discover files and directories without creating Solr documents.
* Properties `Response Handling -> Data ID` is configured to extract unique identifiers from the objects parsed. For most entities, `html_url` provides a globally unique, human-readable URL. For branches and tags, `name` is used since these entities lack `html_url` in list responses.
* Properties `Response Handling -> Parent Data Key` is required by the connector on all child requests. Most children use `parentIdKey=full_name` to extract the repository's `full_name`. The CONTENT and FOLDER requests use `parentIdKey=path` to pass file/directory paths to their child requests via `${LW_PARENT_DATA_KEY}`. The COMMIT request uses `parentIdKey=sha` to pass the commit SHA to its COMMIT_DIFF child request via `${LW_PARENT_DATA_KEY}`.

=== Binary Parsing:
* The BLOB request uses `binaryResponse=true` to enable binary parsing. The request includes the `Accept: application/vnd.github.raw+json` header so the GitHub Contents API endpoint (`/repos/{owner}/{repo}/contents/{path}`) returns raw binary file content instead of a base64-encoded JSON object. With binary response enabled, the connector passes the raw content directly to Apache Tika for text extraction.

== Terminology

The following terms are provided as a reference.

[options="header",cols="1s,1"]
|=======================

|Term|Description
|List of Requests Configuration|Configure List of Requests to extract data from the Rest source. Requests are linked hierarchically by using the properties Parent-Child Request Link -> ObjectType and ParentObjectType.

|Object Type| The unique name to identify the request.
|Parent Object Type| Reference an existent Object Type. Create a parent-child hierarchy, where the current request becomes the child of the specified Parent Object Type. If blank, the current request is considered a Root-Request.

|Root Request|The type of request-configuration to retrieve the initial parent objects.
|Child Request|The type of request-configuration to retrieve children objects per each parent object. A child-request can be a parent of another child-request.

|Response Handling| The responseConfiguration defines the mapping between the response and data objects to be indexed.
|Data Path|The path to access a specific data object within a response. For GitHub endpoints that return bare JSON arrays, set to an empty string. This property accepts JsonPath expressions e.g. `results`, `items[*]`.
|Data ID|The identifier key for the data objects extracted with 'Data Path'. This value will be used to build the solr-document's ID. If not provided, a random UUID will be used. This property accepts JsonPath expressions, e.g. `html_url` to extract the unique URL of an object.
|Parent Data Key|Required for all Child Requests. Map to a key from the parent object, whose value will be used to replace the ${LW_PARENT_DATA_KEY} variable in the child request configuration (endpoint, query params or body). In the repo template, most children set `parentIdKey=full_name` (required by the connector) but hardcode the `{owner}/{repo}` path in the endpoint. The CONTENT and FOLDER requests set `parentIdKey=path` so their child requests receive the file/directory path via `${LW_PARENT_DATA_KEY}` (e.g., `/repos/{owner}/{repo}/contents/${LW_PARENT_DATA_KEY}`).
|_lw_rest_object_type_s| All objects index this field, which value is the 'ObjectType' of the request that retrieved the object, e.g. `REPOSITORY`, `ISSUE`, `PULL_REQUEST`, `COMMIT`.
|_lw_rest_object_s| All objects index this field. Contains the objectId extracted with the property 'Data ID'. E.g.: For a repository, indexes `_lw_rest_object_s: "https://github.com/owner/repo"`. For an issue, indexes `_lw_rest_object_s: "https://github.com/owner/repo/issues/42"`.
|_lw_rest_parent_object_ss| All objects index this field, which value is a list of the objectIds inherited from all their parents, and the objectId from the object itself. E.g.: For an issue, indexes `_lw_rest_parent_object_ss: ["https://github.com/owner/repo", "https://github.com/owner/repo/issues/42"]`.
